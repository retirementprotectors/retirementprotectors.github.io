Shinobi API — Route Group /shinobi

POST /shinobi/message

Receive a Slack message, process it through Shinobi's CCSDK agent (Opus), respond in-channel. This is 2HINOBI's voice when he's not in a live CC session.

Input: { channel_id, user_id, text, thread_ts? }

Output: { success, response_text, actions_taken[] }

POST /shinobi/check

Cron fires every 5 minutes. Reads Slack for unhandled messages. Checks RONIN status. Takes autonomous actions (approve gates, retry transient failures, alert JDM on blockers).

Input: {} (cron-triggered, no body)

Output: { checked_at, slack_messages_processed, ronin_status, actions_taken[] }

POST /shinobi/escalate

RONIN or RAIDEN calls this when blocked or needs strategic guidance. Shinobi (Opus) analyzes the situation and either resolves it or escalates to JDM via DM.

Input: { source: "ronin"|"raiden", sprint_id?, issue, context }

Output: { resolution: "resolved"|"escalated_to_jdm", action_taken, message? }

GET /shinobi/status

What is 2HINOBI/Shinobi doing right now? Returns current state, active tasks, last check time, RONIN status.

Output: { state, active_task?, last_check, ronin_sprint?, uptime }

POST /shinobi/sprint

Queue a new sprint for RONIN. Shinobi validates the discovery doc, seeds tickets if needed, and kicks off RONIN.

Input: { sprint_name, discovery_url }

Output: { sprint_id, tickets_seeded, ronin_started: boolean }

Integration Points

Slack

Read channel C0AP2QL9Z6X. Post responses in-channel. DM JDM (U09BBHTN8F2) at gates and critical escalations. Via callMcpTool('mcp_slack_*').

RONIN

Monitor mdj_forge_runs collection. Auto-resume transient blocks. Queue sprints via direct function calls to forgeRunner (same process — no HTTP overhead).

RAIDEN

/shinobi/escalate endpoint ready for future RAIDEN integration. When RAIDEN can't triage a ticket, it escalates to Shinobi for strategic resolution.

Firestore

mdj_shinobi_state/current singleton doc. Persists: current task, last check time, active sprint monitoring, conversation context. Survives restarts.

gates.ts

When RONIN hits a gate (plan approval, build failure, etc.), gates.ts POSTs to /shinobi/escalate. Shinobi decides: auto-approve (transient CI), or DM JDM (plan/build gates).

Auth

Existing X-MDJ-Auth middleware. Same auth as VOLTRON and RONIN endpoints. No new auth infrastructure needed.

Internal Function Calls, Not HTTP

Shinobi manages RONIN via direct function calls (same Node.js process), not HTTP. No network overhead. No auth round-trips. forgeRunner.resume(), forgeRunner.approve(), forgeRunner.getStatus() — direct imports.
